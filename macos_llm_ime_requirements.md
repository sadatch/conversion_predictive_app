# macOS用 文脈予測強化型 日本語IME 要件定義書

## 1. 背景・目的

macOS標準のライブ変換・予測変換が、直近の入力文脈（数百文字レベル）を十分に活用できておらず、変換候補の質に不満がある。これまでに入力したテキスト（目安：直近500文字程度）を文脈として参照し、より「いい感じ」の変換候補・予測候補を提示する独自の日本語IMEをmacOS上に実装したい。

OS標準のIME（ことえり等）を直接パッチすることはSIPやクローズドソースの制約上現実的でないため、**InputMethodKit (IMKit) を用いた独自IMEをスクラッチで開発し、既存のかな漢字変換エンジンとローカルLLMを組み合わせる**方針で進める。

## 2. システム概要

```
キー入力
  ↓
IMKInputController (Swift, IMKit)
  ↓ ローマ字/かな入力
Mozcサーバー (mozc_server, Unix Domain Socket / Protobuf)
  ↓ 通常の変換候補リスト
ローカルLLM再ランキング・補完層 (llama.cpp)
  ↓ 直近コンテキスト + Mozc候補 → 並べ替え・追加候補生成
IMKCandidates (候補ウィンドウ表示)
```

### 2.1 役割分担の方針
- **かな漢字変換のコア処理（ローマ字パース、活用変化、基本辞書、形態素解析）は自作しない**。Google日本語入力のOSSコアである **Mozc** にそのまま委譲する。
- **「直近の文脈に対してどの候補が最適か」の判断・追加候補生成のみ**をローカルLLMが担当する。LLMはあくまで既存候補のリランキング＋補完に専念し、ゼロから日本語形態素解析をさせない。
- LLM推論はクラウドAPIを使わず、**ローカル実行（llama.cpp）**で完結させる。理由：レイテンシ、プライバシー（入力内容を外部送信しない）、オフライン動作。

## 3. 機能要件

### 3.1 必須機能
1. macOS上で動作するインストール可能なカスタムIME（InputMethodKitベース）であること。
2. ローマ字/かな入力からMozcサーバーへ変換要求を送り、標準的なかな漢字変換候補を取得できること。
3. 直近入力テキスト（目安500文字、可変設定にできるとよい）をバッファとして保持し、変換要求のたびにLLMへ文脈として渡せること。
4. ローカルLLMが、Mozcから得た候補リストを文脈に基づいて再ランキング、および必要に応じて追加候補を生成できること。
5. 最終候補リストをIMKCandidatesのネイティブUIで表示できること。
6. LLM処理が遅延・失敗した場合、Mozc標準候補にフォールバックして変換自体は妨げられないこと（非同期処理、タイムアウト設計）。

### 3.2 非機能要件
- **レイテンシ**: LLMリランキングによる追加遅延は、ユーザー体感で許容できる範囲（目安：数十〜数百ms以内）に収めることを目指す。許容できない場合はMozc標準候補を即時表示し、LLM結果が出たら差し替える非同期UI更新を行う。
- **プライバシー**: 入力内容を外部ネットワークに送信しない（ローカルLLM完結が前提）。
- **安定性**: IME自体のクラッシュや無応答がシステム全体の入力を阻害しないこと（タイムアウト、フォールバック必須）。
- **SIP非依存**: システム保護を無効化したり、Apple非公開APIに依存したりしない範囲で実装する（公式IMKit APIの範囲内）。

## 4. 技術スタック（決定済み事項）

| 項目 | 採用技術 | 備考 |
|---|---|---|
| IME基盤 | InputMethodKit (IMKit) | `IMKInputController`をサブクラス化 |
| かな漢字変換エンジン | Mozc (mozc_server) | Unix domain socket経由、Protobuf (`protocol/commands.proto`) で通信。macOS版Mozcの`mac/`ディレクトリの既存実装を参考にする |
| LLM推論 | llama.cpp | ローカル実行、GGUF量子化モデルを使用 |
| LLMモデル | 未確定（検討中） | 日本語性能と軽量さのバランスを取れるモデル（候補：Sarashina系、Qwen2.5小型版など）を選定する必要あり |
| 実装言語 | Swift（IME本体）＋ Objective-C++（llama.cppブリッジ） | C++ライブラリをSwiftから呼ぶための`.mm`ブリッジが必要 |

## 5. 検討・決定が必要な未確定事項（Opusへの相談ポイント）

1. **Mozcとの通信実装の詳細**
   - `mozc_server`の起動・ライフサイクル管理（IME起動時に自動起動するか、常駐前提か）
   - Protobufスキーマ（`commands.proto`）のうち、変換候補取得に必要な最小のリクエスト/レスポンス構造の特定
   - macOS版Mozc公式実装（`mac/`ディレクトリ）からどこまで流用・参考にできるか

2. **LLMモデル選定とプロンプト設計**
   - リランキングタスクに十分な性能を持つ最小サイズのモデル選定（日本語性能優先）
   - プロンプト設計：直近コンテキスト＋Mozc候補リスト（テキストor簡易JSON）を入力し、並べ替え後のインデックスや追加候補を高速に返す形式
   - 出力フォーマットの安定性（構造化出力、JSON Mode的な制約のかけ方）

3. **非同期処理とUI更新設計**
   - IMKCandidatesは候補表示後の動的差し替えにどこまで対応できるか（API制約の調査）
   - Mozc候補即時表示 → LLM結果到着後に差し替え、というUXが technically 可能か

4. **コンテキストバッファの取得範囲**
   - `client.string(from:)`等でアプリ側テキストフィールドの既存内容を取得できるか（アプリ依存で取得不可なケースの扱い）
   - アプリをまたいだ文脈保持（同一セッション内の複数フィールドなど）が必要か、現時点ではスコープ外とするか

5. **配布・インストール形態**
   - コード署名、Notarization、インストーラーの要否（個人利用前提か配布前提かで変わる）

## 6. スコープ外（現時点）

- カナ漢字変換ロジック自体の自作（Mozcに委譲するため不要）
- クラウドAPI（Claude/GPT等）経由のLLM呼び出し（ローカルLLM完結のため不要、将来的な選択肢として比較検討はあり得る）
- Mozc以外の変換エンジン（SKK等）への対応

## 7. 進め方の想定

1. Mozcソケット通信のPoC（変換候補が取得できることの確認）
2. IMEシェルの作成（Xcodeプロジェクトとして最低限動作する状態まで）
3. llama.cppブリッジ実装とモデル選定・プロンプト設計
4. 上記3要素の統合、非同期フォールバック設計の実装
5. 実利用しながらのチューニング（レイテンシ、候補精度）
