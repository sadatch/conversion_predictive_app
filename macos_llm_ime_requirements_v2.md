## 1. 背景・目的

macOS標準のライブ変換・予測変換は、直近の入力文脈（数百文字レベル）を十分に活用できていない。直近に入力したテキスト（目安：直近500文字程度）を文脈として参照し、より「いい感じ」の変換候補・予測候補を提示する独自の日本語IMEをmacOS上に実装する。

OS標準IME（ことえり等）はSIP・クローズドソースの制約上パッチ不可。そのため **InputMethodKit（IMKit）で独自IMEをスクラッチ実装し、既存のかな漢字変換エンジン（Mozc）とローカルLLM（llama.cpp）を組み合わせる**。

設計の基本姿勢:
- かな漢字変換のコア（ローマ字パース、合成、活用、辞書、形態素解析）は **作らない**。Mozc にそのまま委譲する。
- ローカルLLMは **「直近文脈に対する候補の再ランキング＋少量の追加候補生成」だけ** を担当。ゼロからの日本語解析はさせない。
- LLM推論は **ローカル完結（llama.cpp）**。クラウドAPIは使わない（レイテンシ・プライバシー・オフライン動作）。
- **ディスク使用量を最小化する**（ユーザー最重要要件。§3.3）。

## 2. システム概要

```
キー入力 (NSEvent)
  ↓
IMKInputController (Swift)
  ├─ KeyEvent に変換して Mozc へ
  ↓
Mozc client ライブラリ  (mozc::client::Client / Mach port IPC は内部隠蔽)
  ↓ commands::Output（preedit + candidates）
CandidatePipeline (Swift, 非同期)
  ├─ ① まず Mozc 候補を即時表示（フォールバック兼初期表示）
  ├─ ② 直近コンテキスト(リングバッファ) + Mozc候補 を LLM へ（背景キュー, 期限付き）
  ↓
LLMReranker (Obj-C++ → llama.cpp, GBNFで順位列を強制)
  ↓ 並べ替え後インデックス(+任意で追加候補)
IMKCandidates
  └─ 期限内に返れば初回描画前に並べ替え／間に合わなければ条件付きホットスワップ
```

役割分担:
- **Mozc**: ローマ字/かな入力の解釈、composition、かな漢字変換、候補リスト生成。
- **LLM**: 上記候補を直近文脈に照らして並べ替え。必要なら短い追加候補を提案。
- **IMEシェル**: イベント中継、コンテキスト保持、非同期制御、候補UI表示、フォールバック。

## 3. 機能要件

### 3.1 必須機能
1. macOS上で動作するインストール可能なカスタムIME（InputMethodKitベース、`~/Library/Input Methods/` 配置）。
2. キー入力を Mozc client ライブラリへ渡し、標準のかな漢字変換候補（`commands::Output.candidates`）を取得する。
3. 直近の確定テキストを **インメモリのリングバッファ**（目安500字、設定可変）で保持し、変換要求のたびにLLMへ文脈として渡す。
4. ローカルLLMが、Mozc候補を文脈に基づいて再ランキングし、必要に応じて少数の追加候補を生成する。
5. 最終候補を IMKCandidates のネイティブUIで表示する。
6. LLM処理が遅延・失敗した場合、Mozc標準候補にフォールバック（非同期・タイムアウト）。変換そのものは絶対に阻害しない。**LLM結果が期限内に来た場合のみ並べ替えを反映**。初回描画後のホットスワップは、ユーザーがまだ候補選択を始めていない場合に限る。

### 3.2 非機能要件
- **レイテンシ**: LLMによる追加遅延の目標は体感許容範囲（数十〜数百ms）。許容できない場合はMozc候補を即時表示し、後から非同期で差し替える。LLM推論の **ハード期限（デフォルト150ms、設定可変）** を設け、超過時は今回の並べ替えを破棄。
- **プライバシー**: 入力内容を外部ネットワークに送信しない（ローカル完結）。**ディスクにも書かない**（§3.3）。
- **安定性**: IMEのクラッシュ・無応答がシステム入力を阻害しない。`llama_context` は非スレッドセーフのため **専用シリアルキューで直列化**し、IMKメインスレッドは決してブロックしない。Mozc/LLMいずれの失敗も握りつぶしてフォールバック。
- **SIP非依存**: システム保護の無効化やApple非公開APIへの依存をしない（公式IMKit APIの範囲内）。

### 3.3 ストレージ最小化要件（★最優先）
1. **コンテキストは非永続**: 直近文脈は固定容量のリングバッファとしてメモリ上にのみ保持。ディスクへ保存しない（プライバシーとストレージを同時に満たす）。アプリ終了で消える。
2. **入力ログを残さない**: 変換履歴・入力文字列・LLMプロンプト/応答をファイルやログに出力しない（デバッグ時のみ環境変数で一時的に有効化）。
3. **モデルは単一実体・mmap**: GGUFモデルは `~/Library/Application Support/<IME>/models/` に **1ファイルだけ**置き、llama.cpp の `use_mmap=true` で読み込む（RAM常駐コピーを作らない／アプリ起動を速くする）。`.app` バンドルにモデルを同梱しない（バンドル肥大とコピー重複を回避）。
4. **KV/セッションの永続化なし**: llama.cpp のセッション保存（`state_save`）やKVキャッシュのディスク書き出しはしない。`n_ctx` は控えめ（既定2048）に固定してメモリも節約。
5. **自前キャッシュは最小**: 直近の同一リクエストに対する超短命メモリキャッシュ（LRU 数十件）のみ許容。ディスクキャッシュは持たない。
6. **モデル取得は都度ダウンロード方式**: モデルは配布物に含めず、初回に `scripts/download_model.sh` で取得。再ダウンロード抑止のためのチェックサム確認のみ行い、重複保存はしない。
7. **目標フットプリント**: アプリ本体（.app）は数MB級。総ディスク使用はおおむね「モデル1ファイル（~1GB）＋ Mozc辞書（Mozc側、数十MB）」に収める。

## 4. 技術スタック（決定済み）

| 項目 | 採用技術 | 備考 |
|---|---|---|
| IME基盤 | InputMethodKit | `IMKInputController` サブクラス、`IMKServer`、`IMKCandidates` |
| かな漢字変換 | **Mozc client ライブラリ** | `mozc::client::ClientFactory::NewClient()` を ObjC++ から利用。IPC（Mach port）はライブラリ内部に隠蔽。`commands::KeyEvent` を送信、`commands::Output` を受信 |
| LLM推論 | llama.cpp | ローカル実行、GGUF（Q4_K_M）。`use_mmap=true`、専用シリアルキューで直列化 |
| LLMモデル | **TinySwallow-1.5B-Instruct（Q4_K_M, ~1.0GB）** | 第一候補。代替: Sarashina2.2-1B-instruct（~0.8GB）/ Qwen2.5-1.5B-Instruct（~1.1GB）。超軽量重視なら Qwen2.5-0.5B-Instruct（~0.4GB）も可（§5.2） |
| 実装言語 | Swift（IME本体）＋ Objective-C++（`.mm` ブリッジ） | llama.cpp と Mozc client（いずれもC/C++）を Swift から呼ぶための薄いブリッジ |
| ビルド | Xcode（XcodeGen で `project.yml` から生成） | 混在ターゲット＋Bridging Header。`xcodegen generate` で `.xcodeproj` を生成 |

## 5. 設計詳細（v1の未確定事項に対する回答）

### 5.1 Mozc とのライフサイクル
- **client ライブラリをリンクする方式**を採る。`mozc::client::Client` は内部で `mozc_server`（Converter）プロセスへ Mach port IPC で接続し、未起動なら起動を試みる。したがって IME 側で生のソケット管理は不要。
- 前提として **Mozc 本体（mozc_server）が導入済み**であること（公式OSS Mozcをビルド／インストール、または `mozc_server` をバンドル同梱）。最小構成では「ユーザのMacにMozcを入れておく」運用で可。
- 接続確立は `client->PingServer()` で確認。失敗時はフォールバック（変換は素通り or 何もしない）。

### 5.2 モデル選定（確定）
- **第一候補: TinySwallow-1.5B-Instruct（Q4_K_M, ~1.0GB）** — Qwen2.5-32B からTAID蒸留した日本語特化1.5Bモデル。1GB級で日本語の自然さが高く、リランキング用途に十分。本要件のサイズ/品質目標（~1GB）に最も合致。
- 代替:
  - **Sarashina2.2-1B-instruct（~0.8GB）** — SB Intuitions の日本語モデル。さらに軽量。
  - **Qwen2.5-1.5B-Instruct（~1.1GB）** — 多言語・命令追従が安定。GBNF文法との相性良好。
  - **Qwen2.5-0.5B-Instruct（~0.4GB）** — ストレージを極限まで削るとき。並べ替え精度は要評価。
- モデルは設定で差し替え可能（パスのみ指定）。

### 5.3 プロンプト＆出力フォーマット（リランキング）
- 入力: 直近コンテキスト（末尾N字）＋ Mozc候補（番号付きリスト）。
- 出力: **候補の並べ替えインデックス列のみ**（例 `[2,0,1,3]`）。追加候補を許す場合は別フィールドで短く。
- 出力安定化: llama.cpp の **GBNF grammar** で「角括弧＋整数のカンマ区切り」だけを生成可能にする。これにより不正出力を構文的に排除。`temperature≈0` で決定的に。
- プロンプトは短く保つ（コンテキスト末尾だけ、候補は上位K=最大9件程度）。トークン数を抑え、レイテンシとメモリを節約。

### 5.4 非同期UI更新
- 流れ: ①Mozc候補を即時 `setCandidateData`→`show`。②背景でLLMを期限付き実行。③期限内に並べ替え結果が来たら、**ユーザーがまだ選択操作を始めていなければ** `updateCandidates()`＋`show()` で差し替え。期限超過・失敗時は①のまま。
- 「初回描画前に間に合うならそもそもホットスワップ不要」。期限を短く（既定150ms）すれば、多くは初回描画前に確定し、チラつきを避けられる。
- 追加候補は末尾に append（既存順序を壊さない）。

### 5.5 スレッド／安全設計
- LLM推論は **専用シリアル `DispatchQueue`** で実行（`llama_context` 非スレッドセーフのため）。
- IMKの `candidates(_:)`／`inputText:` などのコールバックは **メインスレッドで即返し**、重い処理は背景キューへ。結果反映はメインスレッドへ戻す。
- 期限管理は `DispatchWorkItem` ＋タイムアウトで実装。期限超過したワークは結果を捨てる（UIに反映しない）。

### 5.6 コンテキスト取得範囲
- 一次ソースは **自前リングバッファ**（確定テキストを `commitComposition` 時に push）。アプリ非依存で常に使える。
- 補助として `client().attributedSubstring(...)` / `string(from:)`（`IMKTextInput`）が使えるアプリでは現在フィールド内容で補強。取得不可アプリでは黙ってリングバッファのみ使用。
- アプリ跨ぎの文脈統合は当面スコープ外（リングバッファはプロセス内グローバルで保持。フィールド切替時の混線は許容、必要なら将来フィールド単位に分離）。

### 5.7 配布・インストール
- 個人利用前提: **ad-hoc 署名**で可。`.app` を `~/Library/Input Methods/` に置き、システム設定→キーボード→入力ソースで有効化。
- 配布する場合のみ Developer ID 署名＋Notarization＋hardened runtime を追加。

## 6. スコープ外（現時点）
- かな漢字変換ロジックの自作（Mozcに委譲）。
- クラウドAPI経由のLLM（ローカル完結。将来比較検討の余地はあり）。
- Mozc以外の変換エンジン（SKK等）。
- アプリ跨ぎ／フィールド単位の高度な文脈管理（将来拡張）。
- LLMによる学習・パーソナライズの永続化（ストレージ最小化方針と相反するため当面なし）。

## 7. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Mozc client ライブラリのビルド・リンクが重い（Bazel） | 統合工数大 | ブリッジを `USE_MOZC` フラグでスタブ化し、まず**スタブで全体をビルド可能**に。Mozc導入後にフラグON（§実装scaffold参照） |
| 小型モデルの日本語リランキング精度 | 候補品質 | TinySwallow-1.5Bを基準に、`temp=0`＋GBNFで安定化。精度不足なら1.5B→3B/差し替え可能に設計 |
| レイテンシのばらつき | 体感悪化 | ハード期限（既定150ms）＋即時フォールバック＋プロンプト短縮＋mmap常駐 |
| IMKCandidates のカスタム差し替えの制約 | UX | 期限短縮で初回描画前確定を狙い、ホットスワップは条件付きに限定 |
| `llama_context` 非スレッドセーフ | クラッシュ | 専用シリアルキューで直列化、メインスレッド非ブロック |

## 8. 進め方
1. IMEシェル＋リングバッファ＋（Mozc/LLMはスタブ）で **ビルド・入力ソース有効化まで** を通す。
2. llama.cpp ブリッジを有効化し、ダミー候補で並べ替えパイプライン（非同期・期限）を確認。
3. Mozc client ブリッジを有効化し、実候補取得を確認。
4. プロンプト/GBNF/期限/コンテキスト長をチューニング（レイテンシ・精度）。
5. 実利用しながら微調整。
